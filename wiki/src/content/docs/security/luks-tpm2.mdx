# Disk Encryption (LUKS + TPM2)

ouroborOS supports optional full-disk encryption via LUKS, with automatic unlocking through TPM2 when available. This provides strong data-at-rest protection without requiring manual passphrase entry on every boot.

## Enabling LUKS

Set `disk.use_luks: true` in your install configuration:

```yaml
disk:
  device: /dev/nvme0n1
  use_luks: true
  btrfs_label: ouroborOS
  swap_type: zram
```

:::caution
LUKS encryption is configured **during installation only**. You cannot enable it retroactively on an existing installation without reinstalling.
:::

## TPM2 Auto-Unlock

When LUKS is enabled and a TPM2 module is present, ouroborOS automatically enrolls the TPM2 chip:

```bash
systemd-cryptenroll --tpm2-pcrs=7+14 /dev/sdX
```

:::note
If no TPM2 module is detected, the system falls back to passphrase-based unlock. You will be prompted for the LUKS passphrase at boot.
:::

## TPM2 PCR Binding Mechanism

The LUKS+TPM2 implementation uses Platform Configuration Registers (PCRs) to bind disk encryption to the boot chain integrity. PCRs 7 and 14 provide comprehensive security coverage:

### PCR 7 (Secure Boot State)
- **Purpose**: Measures UEFI firmware, bootloader signing authority, and Secure Boot configuration
- **Measurement**: Includes Secure Boot keys, firmware modules, and authority chain
- **Protection**: Prevents booting from modified firmware or unsigned bootloaders

### PCR 14 (Boot Entries)
- **Purpose**: Measures systemd-boot configuration and kernel command line
- **Measurement**: Boot menu entries, kernel parameters, and systemd units
- **Protection**: Detects changes to boot configuration or unauthorized boot entries

### PCR Binding Strategy

```bash
# TPM2 enrollment command (executed during installation)
systemd-cryptenroll \
    --tpm2-device=auto \
    --tpm2-pcrs=7+14 \
    /dev/vda2
```

The binding ensures that:
- The LUKS partition can only be unlocked when PCRs 7+14 match their enrolled values
- Any modification to the boot chain invalidates the PCR measurements
- System reverts to passphrase fallback if measurements don't match

## Configuration Options

### Installation Configuration

```yaml
# Enable LUKS encryption with TPM2 auto-unlock
disk:
  device: /dev/nvme0n1
  use_luks: true
  btrfs_label: ouroborOS

security:
  tpm2_unlock: true  # Enable TPM2 auto-unlock for LUKS
```

### PCR Binding Requirements

```bash
# Check TPM2 device availability
ls -la /sys/class/tpm/tpm0

# Verify PCR values before enrollment
tpm2_pcrread sha256:7   # Secure Boot state PCR
tpm2_pcrread sha256:14  # Boot entries PCR

# Verify LUKS device is properly formatted
cryptsetup luksDump /dev/vda2 | grep -i "version"
# Should show:  Version:     2
```

## Post-Installation Management

### Manual TPM2 Enrollment

If TPM2 auto-enrollment failed during installation, you can enroll manually:

```bash
# Using ouroboros-secureboot wrapper
sudo ouroboros-secureboot tpm2-enroll /dev/vda2

# Direct systemd-cryptenroll command
sudo systemd-cryptenroll \
    --tpm2-device=auto \
    --tpm2-pcrs=7+14 \
    /dev/vda2
```

### TPM2 Slot Management

```bash
# Check enrolled TPM2 slots on LUKS device
sudo systemd-cryptenroll --list /dev/vda2

# Remove TPM2 slot (keeps passphrase slot intact)
sudo systemd-cryptenroll --wipe-slot=fido2 /dev/vda2
```

### Status Verification

```bash
# Check PCR values
tpm2_pcrread sha256:7
tpm2_pcrread sha256:14

# Verify TPM2 device status
sudo ouroboros-secureboot tpm2-status

# Check LUKS key slots
cryptsetup luksDump /dev/vda2 | grep -A5 "Tokens"
```

## Fallback Mechanisms

### Passphrase Fallback

The system maintains a passphrase slot alongside the TPM2 slot:

```bash
# Emergency passphrase unlock at boot
# Enter passphrase when TPM2 unlocking fails

# Add additional passphrase post-install
sudo cryptsetup luksAddKey /dev/vda2

# Remove TPM2 slot but keep passphrase
sudo systemd-cryptenroll --wipe-slot=fido2 /dev/vda2
```

### Recovery Scenarios

```bash
# Scenario 1: TPM2 chip failure or replacement
# 1. Boot with passphrase fallback
# 2. Remove old TPM2 slot
sudo systemd-cryptenroll --wipe-slot=fido2 /dev/vda2
# 3. Enroll new TPM2 chip or keep passphrase-only

# Scenario 2: Boot chain modification (PCR mismatch)
# 1. System automatically falls back to passphrase
# 2. Fix boot chain issues
# 3. Re-enroll with new PCR values
sudo systemd-cryptenroll --tpm2-device=auto --tpm2-pcrs=7+14 /dev/vda2

# Scenario 3: Lost TPM2 token (if external)
# 1. Use passphrase fallback
# 2. Remove TPM2 slot
# 3. Configure alternative authentication
```

TPM2 binding with Secure Boot provides layered security:

```bash
# Verify Secure Boot status
mokutil --sb-state

# Check UEFI Secure Boot variables
efivar -l | grep -i secure

# Verify bootloader signatures
sbctl verify /boot/EFI/BOOT/BOOTX64.EFI
```

### Performance Impact


## Integration with Other Security Features

### Combined with FIDO2 Authentication

```yaml
security:
  tpm2_unlock: true        # TPM2 auto-unlock for disk
  fido2_pam: true          # FIDO2 for sudo/login
  secure_boot: true        # Full Secure Boot setup
```

### Dual-Boot Considerations

When dual-booting with Windows:

```yaml
security:
  tpm2_unlock: true
  dual_boot: true         # Enables Windows boot entry detection
  secure_boot: true
  sbctl_include_ms_keys: true  # Required for Windows dual-boot
```

### systemd-homed Integration

For users with systemd-homed LUKS storage:

```yaml
user:
  homed_storage: luks     # LUKS-encrypted home directory
  # Note: Separate from system disk LUKS

security:
  tpm2_unlock: true       # Only applies to system disk, not home directory
```

## Advanced Configuration

### Custom PCR Policies

```bash
# Custom PCR binding (advanced use cases)
sudo systemd-cryptenroll \
    --tpm2-device=auto \
    --tpm2-pcrs=7+14+15 \
    --tpm2-pcr-policy="hash" \
    /dev/vda2

# PCR 15: Measured boot additional event log
```

### Multiple TPM2 Tokens

```bash
# Enroll multiple TPM2 slots for redundancy
sudo systemd-cryptenroll --tpm2-device=auto --tpm2-pcrs=7+14 /dev/vda2
sudo systemd-cryptenroll --tpm2-device=/dev/tpmrm0 --tpm2-pcrs=7+14 /dev/vda2
```

## Troubleshooting

### Common Issues

```bash
# Issue: TPM2 device not found
# Solution:
ls -la /sys/class/tpm/
modprobe tpm
systemctl restart systemd-cryptsetup.service

# Issue: PCR mismatch
# Solution:
tpm2_pcrread sha256:7
tpm2_pcrread sha256:14
# Compare with expected values, fix boot chain

# Issue: systemd-cryptenroll fails
# Solution:
journalctl -u systemd-cryptsetup.service
sudo systemd-cryptenroll --verbose --tpm2-device=auto --tpm2-pcrs=7+14 /dev/vda2
```

### Debug Commands

```bash
# Detailed TPM2 diagnostics
sudo systemd-cryptenroll --tpm2-device=auto --tpm2-pcrs=7+14 --verbose /dev/vda2

# Check cryptsetup service logs
journalctl -u systemd-cryptsetup.service -f

# Monitor boot process
systemd-analyze plot > boot.svg
systemd-analyze critical-chain
```

## Migration and Recovery

### From Passphrase to TPM2

```bash
# 1. Backup existing passphrase
cryptsetup luksAddKey /dev/vda2 backup_key.txt

# 2. Add TPM2 slot while keeping passphrase
sudo systemd-cryptenroll \
    --tpm2-device=auto \
    --tpm2-pcrs=7+14 \
    /dev/vda2

# 3. Verify both slots work
# Test reboot with both TPM2 and passphrase
```

### System Recovery

```bash
# Emergency unlock
cryptsetup luksOpen /dev/vda2 emergency
mount /dev/mapper/emergency /mnt

# Rebuild TPM2 enrollment
sudo systemd-cryptenroll \
    --tpm2-device=auto \
    --tpm2-pcrs=7+14 \
    /dev/vda2

# Verify boot integrity
bootctl
sbctl verify
```
